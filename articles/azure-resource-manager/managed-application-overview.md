# <a name="azure-managed-applications-overview"></a>Azure マネージ アプリケーションの概要

今日 Azure には、ISV や新興企業がそのソリューションを世界中の顧客に提供できる活気に満ちた Marketplace が用意されています。 Azure Marketplace は、ファーストパーティ ベンダーとサードパーティ ベンダーから提供されている、マルチリソースの複合的なテンプレートが何百と集まって構成されているギャラリーです。 顧客は PaaS アプリケーションと SaaS アプリケーションを数分でデプロイして使い始めることができます。 Azure Marketplace は、提供されているソリューションを短時間でデプロイできるすばらしい場ではありますが、ソリューションの保守と更新は、あくまで顧客が行う必要があります。 ベンダーには、アプリケーションの使用に関して、仮想マシン イメージの使用料以外で顧客に課金する術がありません。 また、顧客による重要なアプリケーション リソースの改変を防止したり、アプリケーションの構成要素となっている知的財産権へのアクセスを禁止したりする手立ても、ベンダーにはありません。 Azure マネージ アプリケーションは、こうした問題の解決策となるものです。 

## <a name="advantages-of-managed-applications"></a>マネージ アプリケーションの利点

Azure マネージ アプリケーションは、PaaS サービスまたは SaaS サービスを自己完結型のアプリケーションとしてベンダーが提供できるエコシステムを実現します。 顧客はそのサブスクリプションにマネージ アプリケーションをデプロイすることになりますが、その管理はベンダーが実施できます。 そのため、ベンダーが Azure の課金システムを使って顧客に課金したり、デプロイされたアプリケーションのライフサイクルをテンプレートで管理したりすることが可能です。 顧客側では、更新プログラムを自動的に取得したり、有償のサポートやメンテナンスを利用したりすることができます。 顧客が自らアプリケーションの保守や更新を行う必要はなく、アプリケーションでエラーが発生しても自分で問題を診断して解決する必要もありません。

Azure のこうしたエコシステムは、PaaS ベンダーや SaaS ベンダーだけでなく、各種のソリューションをパッケージして再販売するシステム インテグレーターや企業の要となるプラットフォーム チームにとっても利益となることでしょう。

## <a name="how-managed-applications-work"></a>マネージ アプリケーションのしくみ
マネージ アプリケーションには 2 つの利用形態があります。

1. マネージ アプリケーションを作成して広くユーザーに提供するベンダーまたは独立系ソフトウェア ベンダー (ISV)。 
2. 公開されているアプリケーションを作成して利用する顧客 (コンシューマー)。 

この記事では、両者の利用形態の概要を説明します。 まず、マネージ アプリケーションのしくみを理解しましょう。 

マネージ アプリケーションは、Marketplace のソリューション テンプレートと似ていますが、1 つ大きな違いがあります。 マネージ アプリケーションでは、ISV/ベンダーによって管理されているリソース グループが、リソースのプロビジョニング先となります。 リソース グループは顧客のサブスクリプションに存在しますが、そのリソース グループには、ISV のテナント内のユーザー、ユーザー グループ、アプリケーションがアクセスできます。 アプリケーションの管理や保守を目的として、Active Directory の組み込みのロール (所有者、共同作成者、閲覧者など) にベンダーの ID が追加されます。 

ベンダーから見た利用形態については、「[Azure マネージ アプリケーションの作成と発行](managed-application-publishing.md)」をご覧ください。

コンシューマーから見た利用形態については、「[Azure マネージ アプリケーションの実行](managed-application-consumption.md)」をご覧ください。

## <a name="key-concepts"></a>主要な概念

### <a name="managed-resource-group"></a>マネージ リソース グループ
テンプレートに含まれるすべての Azure リソースのプロビジョニング先となるリソース グループが作成されます。 たとえばストレージ アカウントが作成されるアプライアンスの場合、このリソース グループには、ストレージ アカウント リソースが含まれることになります。 アプライアンス リソースは、ここには含まれません。

### <a name="appliance-package"></a>アプライアンス パッケージ
発行元は、各種テンプレート ファイルと createUIDefinition ファイルを含んだパッケージを作成します。 具体的には、このパッケージには次のファイルが含まれます。

- **applianceMainTemplate.json** - アプライアンスによってプロビジョニングされるすべてのリソースを定義するテンプレート ファイル。 これはリソースの作成に使用される通常のテンプレート ファイルです。

- **MainTemplate.json** - アプライアンス リソース (Microsoft.Solutions/appliances) を定義するテンプレート ファイル。 このリソースに定義されている 1 つの重要なプロパティとして ManagedResourceGroupId があります。 このプロパティは、applianceMainTemplate.json に定義されている実際のリソースが、どのリソース グループを使ってホスティングされるかを示します。

- **createUIDefinition.json** - テンプレートに定義されているパラメーターに必要な UI のレンダリング方法がこのファイルに記述されます。

### <a name="authorization"></a>承認
ベンダーが顧客の代理でリソースを管理するために必要なアクセス許可は、発行元が指定する必要があります。 マネージ リソース グループには、このアクセス許可が適用されます。 次の値を設定します。

- **PrincipalID** - マネージ リソース グループへのアクセスを許可する際に使う、ユーザー、グループ、アプリケーションの Azure AD 識別子。 この識別子は、発行元のテナントに属します。

- **RoleDefinitionID** - 前述のプリンシパル ID に割り当てられるロールの Azure AD 識別子。 発行元のテナントに存在する組み込み RBAC ロールをどれでも使用できます。
