# Machine Learning の Recommendations API の一般的な操作

##目的

このドキュメントでは、サンプル アプリケーションを使用して、Azure ML のいくつかの Recommendations API を使用する方法を示しています。

このアプリケーションは、すべての機能およびすべての API の使用を意図したものではありません。Azure ML の Recommendations サービスを初めて使用する際に実行する一般的な操作の一部を示すだけです。 

##Recommendations の概要

Azure ML Recommendations サービスを使用した推奨機能は、次のデータに基づいて推奨モデルを構築する際に有効です。

* 推奨する項目のリポジトリ (カタログ)
* ユーザーまたはセッションごとの項目の使用状況を表すデータ (これは、サンプル アプリケーションの一部ではなく、データ取得を使用して時間をかけて取得できます)

推奨モデルが構築されたら、それを使用して、ユーザーが選択する一連の項目 (または 1 つの項目) に従って、ユーザーへの推奨項目を予測できます。

このシナリオを使用するには、Azure ML の Recommendations サービスで次の操作を実行します。

* モデルの作成 - これは、データ (カタログと使用状況) と予測モデルを保持する論理コンテナーです。各モデルのコンテナーは、作成時に割り当てられた一意の ID を使用して識別されます。この ID はモデル ID と呼ばれ、API のほとんどで使用されます。 
* カタログのアップロード - モデルのコンテナーが作成されると、それをカタログに関連付けることができます。

注: これらの手順 ("モデルの作成" および "カタログのアップロード") は、通常、モデルのライフ サイクルで 1 回実行されます。

* 使用状況のアップロード - 使用状況データをモデル コンテナーに追加します。
* 推奨モデルの構築 - 必要なデータが取得されたら、推奨モデルのビルドをトリガーします。この操作は、最新の機械学習アルゴリズムを使用して、推奨モデルを作成します。各ビルドが一意の ID に関連付けられています。一部の API の機能に必要になるため、この ID を保持する必要があります。
* 構築プロセスの監視 - 推奨モデルのビルドは非同期操作で、データ (カタログおよび使用状況) の量およびビルド パラメーターに応じて数分から数時間かかります。そのため、ビルドを監視する必要があります。推奨モデルは、関連付けられているビルドが正常に終了した場合のみ構築されます。
* (省略可能) アクティブな推奨モデル ビルドを選択します。この手順は、モデルのコンテナー内に複数の推奨モデル ビルドがある場合のみ必須です。アクティブなビルド (アクティブな推奨モデル) を指定しないで推奨を取得するよう要求すると、システムによって既定のアクティブなビルドに自動的にリダイレクトされます。 

注: アクティブなビルド (推奨モデル) は運用環境で使用でき、運用ワークロード用に構築されます。逆に非アクティブな推奨設定のモデルはテスト環境 (ステージングとも呼ばれます) に留まります。

* 推奨の取得 - 推奨モデルを作成すると、選択した 1 つまたは一連の項目に対する推奨をトリガーすることができます。 

通常は、一定時間の間、推奨の取得を呼び出します。その間、使用状況データを Azure ML の推奨システムにリダイレクトすることができます。システムでは、この使用状況を指定されたコンテナーのモデルに追加します。十分な使用状況データが取得されたら、新しい推奨モデル ビルドをトリガーして、より多くの実際のデータを使用できます。 

##前提条件

* Visual Studio 2013
* インターネットへのアクセス 

##Azure ML のサンプル アプリケーション ソリューション

ソリューションには、ソース コード、サンプルの使用状況とカタログのファイル、およびコンパイルに必要な nuget パッケージをダウンロードするディレクティブが含まれています。

##使用される API

アプリケーションは、使用可能な API のサブセットを通じて、Azure ML Recommendations の一部の機能のみを使用します。次の API がアプリケーションに示されています。

* モデルの作成 - 論理のコンテナーを作成し、データと推奨モデルを保持します。モデルは名前によって識別され、ユーザーは同じ名前のモデルを 2 回作成することはできません。
* カタログ ファイルのアップロード- カタログ データをアップロードします。
* 使用状況ファイルのアップロード - 使用状況データをアップロードします。
* ビルドのトリガー - 推奨モデルを作成します。
* ビルド実行の監視 - 推奨モデル ビルドの状態を監視します。
* 推奨のビルド モデルの選択 - 特定のモデル コンテナーに対して既定で使用する推奨モデルを示します。この手順は、複数の推奨モデルが存在し、非アクティブなビルドをアクティブなビルドとして使用する場合にのみ必要です。
* 推奨の取得 - 与えられた 1 つまたは一連の項目に応じて、推奨項目を取得します。 

API の詳細については、Microsoft Azure Marketplace のドキュメントを確認してください。 

注: モデルには (同時にではなく) 時間の経過と共に複数のビルドが含まれます。各ビルドは、同じまたは更新されたカタログ、および追加の使用状況データで作成されています。

## よくある落とし穴

* サンプルを実行するコマンド ラインとして、ユーザー名と、Microsoft Azure Marketplace のプライマリ アカウント キーを提供する必要があります。
* サンプル アプリケーションを連続して実行すると失敗します。アプリケーション フローは、作成、アップロード、ビルドの監視を処理し、事前定義されたモデル名の推奨を取得するため、呼び出しの間でモデル名を変更しないと、連続した実行は失敗します。
* 推奨はデータなしで終了する可能性があります。サンプル アプリケーションは、ごく小さいカタログと使用状況ファイルを使用するため、作成された推奨モデルには意味がありません。結果としてカタログの項目の一部には推奨項目が返されません。

## 免責事項
サンプル アプリケーションは、運用環境での実行を意図していません。カタログと使用状況に提供されたデータはごく小さく、意味のある推奨モデルとはなりませんが、デモの目的でのみ提供されています。
