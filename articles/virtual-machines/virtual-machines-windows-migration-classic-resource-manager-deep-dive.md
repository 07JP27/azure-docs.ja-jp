# プラットフォームでサポートされているクラシックから Azure Resource Manager への移行に関する技術的な詳細

この記事では、リソースと機能のレベルで移行について技術的に説明します。プラットフォームでリソースがクラシックからリソース マネージャーに移行されるしくみを理解するのに役立ちます。サービス告知記事の「[プラットフォームでサポートされているクラシックから Azure Resource Manager への IaaS リソースの移行](virtual-machines-windows-migration-classic-resource-manager.md)」を参照してください。

## リソースと機能レベルの移行の詳しいガイダンス

以下のリソースはクラシックとリソース マネージャーで呼称が異なる場合があります。この表にない機能とリソースは現時点ではサポートされていません。

| クラシック表示 | リソース マネージャーの表示 | 注記 | | |
|--------------------------------------------------------|-------------------------------------------------|-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|---|---|
| クラウド サービス名 | DNS 名 | 移行中、`<cloudservicename>-migrated` の命名パターンですべてのクラウド サービスに新しいリソース グループが作成されます。これにはすべてのリソースが含まれます。クラウド サービス名は、パブリック IP アドレスが関連付けられた DNS 名になります。 | | |
| 仮想マシン | 仮想マシン | VM 固有のプロパティはそのまま移行されます。コンピューター名など、一部の osProfile 情報はクラシック デプロイメント モデルに保存されておらず、移行後も空のままであることに注意してください。 | | |
| VM に接続されているディスク リソース | VM に接続されている暗黙的なディスク | ディスクは、リソース マネージャー デプロイ モデルの最上位リソースとしてモデル化されません。VM の下で暗黙的なディスクとして移行されます。現時点では、VM に接続されているディスクのみに対応しています。まもなく、関連付けのないディスクを移行できるようになります。移行を有効にするために、クラシック ストレージ アカウントをリソース マネージャー VM で使用できるようになりました。そのため、更新がなくても、ディスクをリソース マネージャー モデルに簡単に移行できます。 | | |
| VM 拡張機能 | VM 拡張機能 | XML 拡張機能を除くすべてのリソース拡張機能が従来のデプロイ モデルから移行されます。 | | |
| 仮想マシン証明書 | Azure Key Vault の証明書 | クラウド サービスにサービス証明書が含まれている場合、プラットフォームはクラウド サービスごとに新しい Azure Key Vault を作成し、証明書をそこに移動させます。Key Vault の証明書を参照し、VM が更新されます。 | | |
| WinRM 構成 | osProfile 下の WinRM 構成 | WinRM 構成は、移行の一環として、そのまま移動されます。 | | |
| 可用性セットのプロパティ | 可用性セットのリソース | クラシック デプロイメント モデルでは、可用性セット仕様は VM のプロパティでした。可用性セットは、移行の一環として、上位のリソースになります。'1 つのクラウド サービスに対して複数の可用性セット' または '1 つまたは複数の可用性セットと、クラウド サービスの可用性セットにない VM' という構成はサポートされていません。 | | |
| VM のネットワーク構成 | プライマリ ネットワーク インターフェイス | 移行後、VM のネットワーク構成がプライマリ ネットワーク インターフェイス リソースとして表示されます。VNET にない VM の場合、内部 IP アドレスが移行中に変わります。 | | |
| VM の複数のネットワーク インターフェイス | ネットワーク インターフェイス | VM に複数のネットワーク インターフェイスが関連付けられている場合、リソース マネージャー デプロイ モデルの移行の一環として、すべてのプロパティと共に、それぞれのネットワーク インターフェイスが上位のリソースになります。 | | |
| 負荷分散されたエンドポイント セット | Load Balancer | クラシック デプロイメント モデルでは、プラットフォームがクラウド サービスごとに暗黙的なロード バランサーを割り当てました。移行中、新しいロード バランサー リソースを作成します。その負荷分散エンドポイント セットがロード バランサー ルールになります。 | | |
| 受信 NAT のルール | 受信 NAT のルール | VM に定義されている入力エンドポイントは、移行中、ロード バランサーの下で受信 NAT ルールに変換されます。 | | |
| VIP または仮想 IP アドレス | DNS 名のあるパブリック IP アドレス | 仮想 IP アドレスまたは VIP がパブリック IP アドレスになり、ロード バランサーに関連付けられます。 | | |
| Virtual Network | Virtual Network | 仮想ネットワークとそのすべてのプロパティがリソース マネージャー デプロイ モデルに移行されます。`-migrated` という名前で新しいリソース グループが作成されます。サポートされていない構成については[ここ](virtual-machines-windows-migration-classic-resource-manager.md)でご確認ください。 | | |
| 予約済み IP | 静的 allocationMethod のあるパブリック IP アドレス | ロード バランサーに関連付けられている予約済み IP がクラウド サービスまたは仮想マシンの移行で移行されます。現時点では、関連付けられていない予約済み IP 移行はサポートされていません。 | | |
| VM ごとのパブリック IP アドレス | 動的 Allocationmethod のあるパブリック IP アドレス | VM に関連付けられているパブリック IP アドレスは、割り当て方法が静的に設定されているパブリック IP アドレス リソースとして変換されます。 | | |
| ネットワーク セキュリティ グループ | ネットワーク セキュリティ グループ | サブネットに関連付けられている NSG は、リソース マネージャー デプロイ モデルへの移行の一環として複製されます。クラシック デプロイメント モデルの NSG は移行中に移動されないことに注意してください。ただし、移行の進行中、NSG の管理プレーン操作は阻止されます。 | | |
| DNS サーバー | DNS サーバー | VNET または VM に関連付けられている DNS サーバーは、すべてのプロパティと共に、該当するリソース移行の一環として移行されます。 | | |
| ユーザー定義のルート | ユーザー定義のルート | サブネットに関連付けられている UDR は、リソース マネージャー デプロイ モデルへの移行の一環として複製されます。クラシック デプロイメント モデルの UDR は移行中に移動されないことに注意してください。ただし、移行の進行中、UDR の管理プレーン操作は阻止されます。 | | |
| VM のネットワーク構成の IP 転送プロパティ | NIC の IP 転送プロパティ | VM上 の IP 転送プロパティは、移行中、ネットワーク インターフェイスでプロパティに変換されます。 | | |
| 複数の IP のあるロード バランサー | 複数のパブリック IP リソースのあるロード バランサー | ロード バランサーが関連付けられているパブリック IP はすべてパブリック IP リソースに変換され、移行後、ロード バランサーに関連付けられます。 | | |
| VM の内部 DNS 名 | NIC の内部 DNS 名 | 移行中、リソース マネージャー デプロイ モデルのサフィックスに合わせて内部 DNS サフィックスが更新されます。移行後、引き続き名前解決が動作することに注意してください。ただし、VM に関連付けられているサフィックスは変更されます。 | | |
