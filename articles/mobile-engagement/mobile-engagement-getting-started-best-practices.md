## <a name="step-1-define-your-kpis-to-fit-the-bet-model"></a>手順 1: おすすめコンテンツ モデルに合わせて、Kpi を定義します。
Kpi を正しく定義すると、完了することは難しい作業を指定できます。 さまざまな業種用に設計されたアプリは、独自の詳細および目標があります。 実際のアプローチを混乱させるこの傾向があります可能性。 これを回避するためには、目標および Kpi を 3 つのカテゴリに分類する必要があります:**ビジネス**、 **Engagement**、および**テクニカル**です。 これと呼ばれるもの、**おすすめコンテンツ モデル**です。

ビジネス Kpi には、作成する最も簡単な含める必要があります。 おそらく既に定義して何らかの形でこれらモバイル アプリを計画するとき。 これらの Kpi では、アプリのメジャーの収益と ROI を一般に役立ちます。 次の一覧は、いくつかの例に従って、業績評価指標を定義する際に助けとなるビジネス Kpi を示します。

* メディア ビジネス Kpi

  * 現在のサブスクリプションの数
* ゲーム ビジネス Kpi
  * アプリ内購入数
  * ユーザー (ARPU) あたりの平均収益
  * セッションごとに費やされた時間
  * 試合レベルで再生し、現在の日
* E コマース ビジネス Kpi
  * 使用している日数アプリ
  * ユーザー (ARPU) あたりの平均収益
  * チェック アウト時にカート内の平均値
  * ほとんどのビューと購入の製品カテゴリ
* 銀行、保険のビジネス Kpi

## <a name="step-3-app-integration"></a>手順 3: アプリの統合
#### <a name="create-a-tag-plan"></a>タグのプランを作成します。
Azure Mobile Engagement をアプリに統合するには、タグ プランを作成する必要があります。 タグ プランは、プロジェクトの基礎となります。 マーケティング仕様をアプリケーションの作業の流れと Kpi を測定するアプリで収集された実際のタグ データ間のリレーションシップを定義します。 これは、どのような分析ことができます、ポータルで表示することを示します。 ユーザー セグメントを定義し、エンドユーザーが参加する対象を絞ったプッシュ通知を送信することができます。 1 回追加アプリに統合するためのコードは、Azure Mobile Engagement の SDK を使用して単純化、定義されているタグ計画があります。

タグ プランでは、アプリケーションですべてのタグが付けしない必要があります。 これにより、モバイル エンゲージメント戦略の一部であるタグ データがのみ含まれます。 これには、アプリケーション間でさまざまな可能性がありますになります。 [メディア戦略テンプレート][ Media Playbook link]して Azure Mobile Engagement により指定されたメソッドを使用してタグ プランをビルドして提供します。 使用して、**タグ プラン**タグをプラン ガイドとワークシートです。

ワークシートでタグに関するセクションを定義する、非常に限定します。 これは、混乱を避けるために非常に重要です。 各詳細では、各タグが送信されるシナリオが必要です。 各タグが埋め込まれているアクティビティの名前が含まれます。 これは、すべてに含めるか、**情報**ワークシートの一部です。 タグの計画ワークシートでは、テストの検証の主要な参照をする必要があります。 

**データを収集する**セクションで、開発チームはタグの種類、名前、値、および各に必要な追加情報のキー/値ペアを検索に埋め込まれますアプリケーションです。

すべてのチーム プロジェクトに関連付けられているタグ計画を見直すことをお勧めします。 必要な修正を行い、マーケティングと開発チームに明確ではすべてのものを確認します。

**作業明細書**ワークシートを使用して、プロジェクトに関連するすべてのユーザー ガイドを支援することです。

#### <a name="data-types"></a>データ型
これらは、一般的な Azure Mobile Engagement によってデータのサポートの種類です。

###### <a name="devices-and-users"></a>デバイスとユーザー
Azure Mobile Engagement は、各デバイスの一意の識別子を生成することによって、ユーザーを識別します。 デバイス識別子 (または deviceid) は、この識別子が呼び出されます。 同じデバイスで実行されているすべてのアプリケーションが同じデバイス id を共有するように生成されます。

###### <a name="sessions-and-activities"></a>セッションと活動
セッションは、ユーザーが実行されているアプリの 1 つのインスタンスです。 セッションを停止するまで、ユーザーが、アプリを開始時刻からスパンします。

アクティビティは、作業をアプリがセッション中に行うことがありますが、一連の論理グループです。 これは通常、アプリで、特定の画面がアプリケーションのロジックによって定義されたものであることができます。 少なくともは、アプリの各画面またはアクティビティ タグ付けする必要があります。 こうと、ユーザーのパスを理解します。

###### <a name="events"></a>イベント
アプリでのユーザー操作を報告するイベントが使用されます。 インスタント アクションは、コンテンツを共有またはビデオを起動するようにできます。 イベントのタグ付けがユーザーがアプリと対話する方法を表示するデータ コレクションに提供されます。 

###### <a name="jobs"></a>Jobs
ジョブは、レポート、存続期間があるアクションに使用されます。 いくつかの例を含めます。

* API 呼び出しの実行
* 広告の表示時間
* バック グラウンド タスクの継続時間 
* 発注プロセスの期間
* ビデオを表示します。

###### <a name="errors"></a>エラー
エラーは、アプリによって検出された問題の報告に使用されます。 たとえば、誤ったユーザー操作または API 呼び出しに失敗したします。

###### <a name="application-information"></a>アプリケーションの情報
アプリケーション情報 (App-info) は、アプリケーションのユーザーのエクスペリエンスに関連するデータにタグを付けるために使用されます。 アプリケーションでのユーザーの操作によって生成されます。 

アプリ情報の指定したキーの Azure Mobile Engagement のみの追跡、最新の値 (履歴がありません)。 アプリ情報では、アプリまたはエンドユーザーの状態が表示されます。 たとえば、ログの状態、またはユーザーのお気に入りの製品のグループ。

###### <a name="crash-data"></a>クラッシュ データ
アプリケーションで処理されない、Mobile Engagement SDK レポート アプリケーションの障害によって自動的に収集されたデータがクラッシュします。 たとえばに発生する未処理の例外です。

###### <a name="extra-data"></a>余分なデータ
パラメーターを使用して、イベント、エラー、アクティビティ、およびジョブを拡張することができます。 これは、余分な情報が、開発者がアプリケーションから特定のデータとして提供可能性があります。 これは粒度の細かいセグメント化を定義するために重要です。 

たとえば、「記事」タグの値、その特定のアーティクルを表示したに基づいてセグメント エンドユーザーにできます。 十分でないです。 その同じ「記事」タグには、アクティビティ内で"news_category"などの追加情報も含まれている場合、改善が必要があります。 ユーザーのお気に入りのカテゴリを動的に判断すると便利になります。 

余分な情報については、キー/値のペアとして報告されます。 このメディア アプリケーションの例では、"news_category"の余分な情報をそのカテゴリの値となります。 たとえば、「スポーツ」、"economy"または「政治」にします。

## <a name="conclusion"></a>結論
エンゲージメント プログラムは、反復的なとするアプリが最適に動作を試すとき自分継続的に改善する必要があります。 

全体のグローバル エンゲージメント戦略を構築しようとしてしないで戦略 engagement エクスペリエンスを開発中に、最初に、します。 Kpi、およびそれらを活用する方法を識別する手順の方法で行います。 エンゲージメント戦略は、各アプリに対して一意であるされます。

いくつかのエクスペリエンスを開発した後、engagement プログラムに、次を追加するよう考慮します。

* 追跡: は、ユーザーを取得して、データ コレクションのソースを定義する可能性があります。 Azure mobile Engagement は、データ コレクションのソースにリンクすることができます。 各ソースのパフォーマンスを監視できます。 この情報は、購入への投資を最大化にとって重要となります。 
* A B のテストを実行: Engagement プログラムの重要な要素です。 アプリごとに、独自の仕様です。 With A/テスト B、engagement プログラムを向上できます。
* 地理的な場所: これは、ブランドの大規模な営業案件です。 この機能のおかげで、適切な場所と時間に到達できます。 地理的な場所の使用を開始する前に十分なエンドユーザーの動作のデータを収集したを確認することをお勧めします。
* データ プッシュ。 データ プッシュは非表示のプッシュします。 データ プッシュは、エンドユーザーの動作に基づいて、アプリケーションをカスタマイズできます。 たとえば、多くの場合、ユーザー セグメントには、ハイテク製品が参照して、アプリ所有者は、ハイテク コンテンツで自分のホーム ページをカスタマイズするデータのプッシュを送信できます。
